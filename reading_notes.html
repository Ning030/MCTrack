第一层：项目地图

1.1 项目概览

MCTrack 是由旷视研究院（megvii-research）开发的统一3D多目标跟踪（MOT）框架。 该项目在 KITTI、nuScenes、Waymo 三大自动驾驶数据集上均达到 SOTA 水平，论文已被 IROS 2025 接收。 核心技术栈为 Python + NumPy + 扩展卡尔曼滤波（EKF）+ 匈牙利/贪心匹配算法，采用多进程并行处理每个场景。 项目提出了一种名为 BaseVersion 的统一数据格式，将不同数据集的检测结果转换为标准格式，使研究者可以专注于跟踪算法本身。

3.2 核心业务逻辑

流程一：数据预处理（BaseVersion 生成）

🔄 坐标转换流水线 preprocess/

将各数据集的原始检测结果统一转换为 BaseVersion JSON 格式：

  1. 读取标定参数：Camera 内参（P2）、LiDAR→Camera 外参（Tr_velo_to_cam）
  2. 读取自车位姿：4×4 变换矩阵，用于 LiDAR→全局坐标转换
  3. 坐标转换链：Camera 3D → LiDAR → Global（使用齐次坐标变换）
  4. 输出：统一的 JSON 格式，包含 global_xyz, global_yaw, lwh, velocity 等字段

📁 preprocess/convert_kitti.py L238-296 — convert_baseversion_bboxes()

流程二：单帧跟踪（核心管线）

🎯 两阶段匹配策略 tracker/base_tracker.py L33-138

第一阶段（BEV 匹配）：在鸟瞰视角（Bird's Eye View）下使用 RO-GDIOU-3D 代价函数计算跟踪轨迹与检测的关联代价矩阵， 通过匈牙利算法或贪心算法求解最优分配。

第二阶段（RV 匹配，可选）：对第一阶段未匹配的轨迹和检测，在重投影视图（Re-projected View）下使用 SDIOU-2D 代价函数进行二次匹配。 额外约束：旋转差 < 90° 且距离 < 5m（L108）。

流程三：轨迹生命周期管理

🔄 轨迹状态机 tracker/trajectory.py L76
状态定义：
  0: initialization (初始化)    ← 新轨迹创建时
  1: confirmed    (已确认)      ← track_length > confirmed_length 或匹配分数足够高
  2: obscured     (被遮挡)      ← confirmed 后连续 unmatch > max_unmatch
  4: dead         (已终止)      ← obscured 后连续 unmatch > max_predict

状态转换图：
  [新检测] → 0(init) ─── 匹配足够多帧 ──→ 1(confirmed)
                                              │
                                    连续未匹配 > max_unmatch
                                              │
                                              ▼
                                         2(obscured)
                                              │
                                    连续未匹配 > max_predict
                                              │
                                              ▼
                                          4(dead) → 移入 all_dead_trajs

流程四：扩展卡尔曼滤波

🔬 多运动模型滤波 kalmanfilter/extend_kalman.py

每条轨迹同时维护 5 个独立的卡尔曼滤波器，根据配置选择主运动模型：

滤波器状态维度状态向量用途
EKF_CV4D[x, y, vx, vy]匀速运动（Constant Velocity）
EKF_CA6D[x, y, vx, vy, ax, ay]匀加速运动（Constant Acceleration）
EKF_CTRA6D[x, y, ψ, v, ω, a]恒定转弯率加速度（最复杂的模型）
KF_YAW2D[yaw, yaw_rate]航向角估计
KF_SIZE4D[L, W, dL, dW]尺寸（长宽）估计

滤波基类采用 模板方法模式：KF_Base 定义 update() 和 predict() 骨架， 子类实现 f(x)（状态转移）和 getF(x)（雅可比矩阵）。

📁 kalmanfilter/base_kalman.py L51-67 — update() 实现标准 EKF 预测-更新循环

流程五：代价函数计算

📐 RO-GDIOU-3D 代价 tracker/cost_function.py L34-97

核心的 BEV 匹配代价函数 cal_rotation_gdiou_inbev() 计算公式：

RO-GDIOU = IoU − w1 × (extra_area / universe_area) − w2 × (center_distance / diagonal_distance)

其中：
  IoU        : 使用 cv2.rotatedRectangleIntersection() 计算旋转矩形交集
  extra_area : 最小外接矩形面积 − 两个框面积并集
  center_dist: 两框中心欧氏距离的平方
  w1, w2     : 可配置的权重参数（按类别区分）

该公式综合考虑了 重叠度（IoU）、空间紧凑性（extra_area 惩罚项）和 中心距离（center 惩罚项）， 比标准 IoU 更适合旋转目标的关联。

④ EKF_CTRA — 最复杂的运动模型 kalmanfilter/extend_kalman.py L289-449

作用

恒定转弯率加速度（Constant Turn Rate and Acceleration）扩展卡尔曼滤波器，用于建模弯道行驶等非线性运动。

状态向量

[x, y, ψ(yaw), v(速度), ω(角速度), a(加速度)] — 6维状态

状态转移方程（L309-328）

# 当 ω ≠ 0（转弯）时：
x_new = x + (1/ω²) × [(v×ω + a×ω×dt)×sin(ψ+ω×dt)
                      + a×cos(ψ+ω×dt) - v×ω×sin(ψ) - a×cos(ψ)]
y_new = y + (1/ω²) × [(-v×ω - a×ω×dt)×cos(ψ+ω×dt)
                      + a×sin(ψ+ω×dt) + v×ω×cos(ψ) - a×sin(ψ)]
ψ_new = ψ + ω×dt
v_new = v + a×dt

# 当 ω ≈ 0（直行）时：退化为恒定加速度模型

雅可比矩阵（L330-423）：6×6 矩阵，包含对 ψ, v, ω, a 的偏导数，实现精确线性化。

⑧ KF_Base.update() kalmanfilter/base_kalman.py L51-67

EKF 预测-更新算法实现

def update(self, z):
    # ─── 预测步骤 ───
    self.x = self.f(self.x)           # 状态转移：x̂ = f(x)
    self.F = self.getF(self.x)        # 获取雅可比矩阵 F
    self.P = F × P × F^T + Q         # 协方差预测

    # ─── 更新步骤 ───
    G = P × H^T × (H×P×H^T + R)^-1  # 卡尔曼增益
    self.x += G × (z − h(x))         # 状态校正（测量残差加权）
    self.P = (I − G×H) × P           # 协方差更新

    return x.reshape(n)               # 返回更新后状态

注意：预测和更新合并在同一个 update() 调用中（L56-67），这意味着每次 update() 调用都会先执行一次预测。 单独的 predict() 方法（L69-73）仅返回预测值，不改变内部状态（但实际上 f(x) 可能有副作用）。

总结与洞察

💡 亮点设计

  1. BaseVersion 统一格式：将 KITTI/nuScenes/Waymo 三个差异巨大的数据集统一为相同的 JSON 格式， 极大降低了跟踪算法的数据适配成本。研究者只需关注跟踪逻辑本身。
  2. 两阶段匹配策略：先 BEV（3D 鸟瞰）匹配主体目标，再 RV（2D 相机重投影）匹配遗漏目标， 利用互补视角提升召回率，同时通过旋转角和距离约束避免误匹配。
  3. 多运动模型并行：每条轨迹同时初始化 CV/CA/CTRA 三种运动模型，通过配置为不同类别选择最适合的模型 （如：行人用 CV、车辆用 CTRA），兼顾通用性与精度。
  4. RO-GDIOU 代价函数：在标准旋转 IoU 基础上引入空间紧凑性和中心距离惩罚， 对远距离、低重叠的目标提供更好的关联区分度（cost_function.py L34-97）。
  5. BackPredict 机制：利用检测结果的速度反向预测到上一帧时刻（bbox.py L54-56）， 与轨迹融合态做匹配，补偿预测误差。可通过配置选择 Predict/BackPredict/Fusion 三种代价计算状态。

⚠️ 潜在问题

  1. KF update() 副作用：KF_Base.update()（L51-67）将预测和更新合并在一个方法中， 调用一次 update 会自动推进一步预测。在 unmatch_update()（trajectory.py L246-254）中， 先调用 predict() 再调用 update(predict_state)，相当于执行了两次预测步骤， 这可能导致协方差过度膨胀。
  2. 缺少单元测试：项目没有任何测试文件（无 tests/ 目录、无 *_test.py）， 关键的 EKF 数值正确性和匹配算法边界条件缺乏验证。
  3. 无 CI/CD：缺少自动化测试和构建流程，代码质量依赖人工审查。
  4. 硬编码约束：RV 匹配中的旋转差阈值 90° 和距离阈值 5m 是硬编码的 （base_tracker.py L108），不同场景可能需要不同的值。
  5. 内存管理：_cache_bbox_len 控制轨迹最大缓存长度（通过 pop(0) 移除旧 BBox）， 在 Python list 上 pop(0) 是 O(n) 操作，使用 collections.deque 会更高效。

📚 值得学习的地方

  1. 清晰的分层架构：代码组织为 dataset → tracker → evaluation 三层， kalmanfilter 和 utils 作为独立工具模块，依赖关系单向流动，易于理解和扩展。
  2. 配置驱动设计：几乎所有超参数（匹配模式、代价函数、卡尔曼噪声、阈值）都通过 YAML 配置管理， 每个类别可以有不同的参数配置，切换数据集只需更换配置文件。
  3. 多进程架构：利用 multiprocessing.Pool 并行处理独立场景， 使用 Manager().dict() 共享结果，简洁高效（main.py L145-157）。
  4. 丰富的代码注释：关键函数都有详细的 docstring，说明参数含义和返回值格式， 降低了代码阅读门槛。
  5. EKF 实现教科书级别：KF_Base + 5 种子类的实现结构清晰， 特别是 EKF_CTRA 的复杂非线性状态转移和雅可比矩阵推导值得深入学习。
